## 项目快速定位

- **一句话总结：** KOB 是一个支持真人与 Java Bot 实时对战的平台，计划升级为能够自动生成、评测和改进 Bot 的有界自主 Agent 系统。
- **解决的问题：** 原项目解决在线匹配、实时裁判和用户代码执行；Agent 化后进一步解决策略生成缺乏反馈、代码质量无法自动验证和迭代过程不可观测的问题。
- **目标用户：** 算法学习者、Bot 策略开发者，以及希望观察 Agent 自动改进过程的开发者。
- **技术领域：** Java 后端、WebSocket、并发、代码沙箱、确定性评测、Agent Workflow、Tool Calling。
- **同类项目：** 在线判题或 Bot 竞技平台、游戏 AI Arena、代码生成 Agent、自动化评测平台。
- **面试价值：** 强于普通 CRUD 和聊天套壳项目，能同时覆盖实时系统、沙箱、协议可靠性、Agent 状态机和 LLM 评测。

## 当前架构概览

```text
Web 前端
├── REST + JWT -> backend
└── WebSocket -> backend -> Game 线程
                         ├── matching 回调创建对局
                         ├── botrunning 执行 Bot
                         ├── 推送 move/result
                         └── 事务化保存积分与录像
```

当前 3 个后端模块：

| 模块 | 作用 | 关键入口 |
| --- | --- | --- |
| `backend` | 用户、Bot、实时对战、录像、排行榜 | `BackendApplication.java` |
| `matchingsystem` | 积分与等待时间匹配 | `MatchingPool.java` |
| `botrunningsystem` | 动态编译与隔离执行 Java Bot | `BotPool.java`、`BotExecutor.java` |

- **证据：** `BotExecutor.java`、`ProcessSandboxBotExecutor.java:35-105`、`SandboxMain.java:21-65`。
- **最小复现：** 保留独立 JVM、执行超时和临时目录，增加恶意代码测试。
- **包装路径：** 如果这批重构由你完成，可以直接作为当前项目亮点。
- **面试风险：** 必须承认 Java 8 `SecurityManager` 不是生产级容器隔离。

#### 2. `gameId + roundId` 请求关联协议

- **证据：** `Game.java:30-31`、`Game.java:84-102`、`Game.java:196-197`。

## 核心架构要点

- **当前数据流：** 前端通过 WebSocket 发起匹配和移动，`WebSocketServer` 调用匹配服务，匹配回调后创建 `Game`；Bot 移动经 `botrunningsystem` 执行并回调，最终事务化保存积分和录像。
- **当前核心调用链：** `WebSocketServer.startGame()` 创建 `Game` 并启动线程；`Game.nextStep()` 请求 Bot；`Consumer` 调用 `BotExecutor`；回调由 `Game.applyBotMove()` 校验对局与回合。
- **计划数据流：** Agent 任务进入 `AgentWorkflowService`，依次调用生成、编译和批量评测工具；公开集结果进入下一轮模型上下文，隐藏集只在全部迭代结束后参与版本选择。
- **模块拆分：** `backend` 负责编排和持久化，`botrunningsystem` 负责不可信代码执行，新增 `game-core` 负责确定性规则，`web` 负责 Trace 与版本展示。
- **关键模式：** Strategy、Executor、状态机、生产者-消费者、事务服务、注册表、发布器和受控 Tool Router。
- **架构优点：** 已有安全与正确性重构可以直接支撑 Agent 化，不需要推翻现有系统。
- **架构风险：** Java 8 与 `SecurityManager` 较旧；`Game` 和 `WebSocketServer` 仍有线程与静态依赖债务；首版只能定位为本地受控演示。

## 关键技术点

### 1. Bot 请求关联协议

- **解决的问题：** Bot 回调可能迟到、重复或乱序，仅凭 `userId` 会把旧结果写入新对局。
- **源码证据：** `Game.java:30-31` 生成 `gameId` 和 `currentRoundId`；`Game.java:84-102` 在应用回调前校验；`Game.java:196-197` 将关联字段传给执行服务。
- **项目本质：** 这是异步系统中的请求关联与过期响应隔离，不只是游戏字段补充。
- **面试怎么讲：** 「我为 Bot 调用增加对局级和回合级关联标识，旧回调即使到达也无法推进当前状态。」
- **容易被追问：** 为什么不只用 `requestId`；重复回调是否幂等；并发回调如何保证可见性。

### 2. 进程级 Bot 沙箱

- **解决的问题：** 用户代码不能在主 JVM 内无限循环、联网、写文件或执行命令。
- **源码证据：** `ProcessSandboxBotExecutor.java:35-93` 创建临时目录和独立 JVM，超时后强杀；`SandboxMain.java:21-65` 安装权限限制并动态编译执行。
- **项目本质：** 将不可信代码执行与业务服务隔离，并通过执行器接口保留切换能力。
- **面试怎么讲：** 「我先抽象 `BotExecutor`，再用独立 JVM 实现超时强杀和文件隔离，避免线程中断无法终止恶意代码。」
- **容易被追问：** `SecurityManager` 的局限；CPU、内存和进程数如何限制；为什么不是 Docker。

### 3. Bot 队列并发边界

- **解决的问题：** 旧实现存在锁计数泄漏，并且耗时消费逻辑可能阻塞生产者。
- **源码证据：** `BotPool.java:14-21` 在锁内入队并唤醒；`BotPool.java:29-48` 在锁内取任务、锁外执行。
- **项目本质：** 临界区只保护队列状态，执行用户代码不占用队列锁。
- **面试怎么讲：** 「生产消费只在队列操作时持锁，动态编译和执行移动到锁外，避免整个 Bot 服务被单个慢任务拖住。」
- **容易被追问：** 为什么使用 `Condition` 而不是 `BlockingQueue`；如何限制 Consumer 数量；当前每任务建线程的风险。

### 4. 实时对战一致性

- **解决的问题：** 浏览器后台限帧、Bot 快速回调和 WebSocket 重连可能让双方消费不同移动序列。
- **证据：** 提交 `09138ea` 增加按回合排队、协议顺序约束、连接实例安全移除和 Playwright 三类对战验证。
- **项目本质：** 服务端事件顺序与客户端渲染节奏必须解耦，最终棋盘状态应由消息序列决定。
- **面试怎么讲：** 「我没有继续加固定延时，而是调整消息消费和重绘规则，使后台标签页和快速 Bot 仍能收敛到同一状态。」
- **容易被追问：** 网络乱序如何处理；多节点 WebSocket 怎么路由；客户端重放如何校验。

### 5. 确定性 Agent 评测闭环

- **状态：** 设计已确认，尚未实现。
- **解决的问题：** LLM 生成代码后无法仅凭文本判断是否真正变强。
- **设计证据：** `docs/superpowers/specs/2026-07-16-kob-agent-lab-design.md`。
- **项目本质：** 把模型能力评估下沉到工程系统，通过固定种子、基准 Bot、隐藏集和版本拒绝规则形成可重复实验。
- **面试怎么讲：** 完成实现后可说「模型只负责提出候选策略，是否提升由确定性评测器决定，隐藏集负责阻止针对公开地图刷分。」
- **容易被追问：** 指标是否公平；隐藏集是否真的不可见；如何证明提升不是随机波动。

## 亮点选择建议

### 优先实现

- **有界 Agent Workflow + 自动评测闭环**
  原因：这是项目从「旧课程项目」变成「Agent 时代项目」的核心。
  最小实现：V1、V2 两轮，`FakeLlmClient`，1 个基准 Bot，固定地图。
  完整实现：V1～V3、3 个基准 Bot、迭代后隐藏验证、Trace 和版本拒绝。

- **`game-core` 与批量评测沙箱**
  原因：既证明 Java 领域建模，也解决评测性能和安全问题。
  最小实现：抽出地图与回合推进，运行固定策略对战。
  完整实现：在线与离线共用规则，隔离进程内编译一次并连续评测。

- **持久化状态机与幂等 Step**
  原因：与真实 Agent 平台最接近，能连接已有乐观锁评测经验。
  最小实现：状态落库、唯一幂等键、取消。
  完整实现：重启恢复、Token 和耗时追踪。

### 可顺手完成

- 代表性失败录像选择。
- 版本代码 Diff。
- Agent Token、耗时和工具成功率统计。
- 最佳版本一键保存到现有 Bot。

### 暂不投入

- 多 Agent 辩论。
- 公网部署。
- 分布式任务队列。
- 大版本技术栈升级。
- 实时 LLM 走棋。

这些方向会扩大工程量，但不会显著增强首版面试故事。

  **诚信边界：** 当前积分更新仍需评估高并发丢失更新风险，不要扩展成「解决全部并发一致性」。

### 完成 Agent Lab 后可写

- 在实时 Bot 对战平台上设计并实现有界自主 Agent，通过代码生成、沙箱编译、批量评测和失败分析工具完成最多 3 轮策略自优化，并基于隐藏评测集自动选择最佳版本。
  **需要先完成：** Workflow、Tool Router、V1～V3、隐藏集和端到端演示。
  **指标边界：** 只有真实实验完成后才能填写胜率提升。

- 抽取确定性 `game-core` 规则引擎，复用于在线裁判与离线评测；设计固定种子、双边出生点和多基准 Bot 的评测协议，将综合胜率从真实基线提升至真实结果。
  **需要先完成：** 规则抽取、回归测试、72 局评测和可复现实验脚本。
  **指标边界：** 不得提前填写 `X% → Y%`。

- 基于持久化状态机、数据库乐观锁和幂等工具调用实现 Agent 长任务恢复与取消，记录版本 Diff、Agent Trace、Token 和延迟指标。
  **需要先完成：** 4 张表、状态机、唯一幂等键、重启恢复和 Trace 页面。
  **面试可讲：** 为什么没有直接引入消息队列，以及达到什么规模后再拆 Worker。

### 只适合面试谈资

- 为什么 MVP 不新增第 4 个微服务：本地任务规模有限，编排和用户数据强相关，先用包边界与接口隔离，等容量证据出现再拆。
- 为什么不使用 LLM 实时走棋：延迟、成本和非确定性会污染实时对战；LLM 更适合离线生成与诊断。
- 为什么 `SecurityManager` 不是长期方案：生产环境需要容器、cgroup、seccomp 或独立执行节点。

## 30 秒介绍

KOB 原本是一个 Java 实时 Bot 对战平台。我先对旧系统做了安全和正确性重构，包括 Bot 进程沙箱、异步回调关联协议、事务化结算和 WebSocket 状态一致性。下一步我把它升级成 KOB Agent Lab：模型负责生成和诊断策略，确定性游戏引擎与沙箱负责执行和评测。系统会自动跑固定基准赛，根据失败录像迭代代码，并用隐藏评测集防止过拟合。

## 2 分钟介绍

KOB 最初支持用户提交 Java Bot，与真人或其他 Bot 进行实时贪吃蛇对战。系统由主后端、匹配服务和 Bot 执行服务组成，使用 WebSocket 推送每回合移动和结算。

我前段时间先处理了旧系统最危险的部分。原来用户代码在主 JVM 中动态执行，线程中断无法可靠停止死循环，所以我抽象了 `BotExecutor`，增加独立 JVM 沙箱、临时目录和超时强杀。异步 Bot 回调原来只有 `userId`，容易把旧回调写入新对局，因此增加了 `gameId` 和 `roundId`。结算部分则抽成事务服务，保证积分和录像原子保存。

Agent 化改造不会简单增加聊天框。我计划抽出确定性的 `game-core`，让在线裁判与离线评测共用规则。用户输入自然语言目标后，Agent 只能调用生成代码、编译、批量评测和改进代码等受控工具，最多迭代 3 次。公开评测结果可以反馈给模型；隐藏评测集只在全部迭代结束后运行，用于最终版本选择。这样模型负责创造候选方案，系统负责验证事实。

项目最终要证明的不只是「模型能写代码」，而是 Agent 在安全边界、失败恢复和自动评测约束下，能否稳定地产生更好的可执行策略。

## 深度讲解版本

这个项目的复杂度中心有两个。

第一个是实时对战链路。服务端需要同时处理玩家 WebSocket、匹配服务回调和 Bot 执行回调。不同来源的事件有不同延迟，不能依赖固定等待时间。当前实现通过回合状态、请求关联和连接实例安全移除，使旧事件不能污染新连接或新回合。

第二个是不可信代码执行。仅使用线程超时无法停止忽略中断的代码，因此需要进程边界。现有实现通过 `ProcessBuilder` 启动独立 JVM、使用临时目录隔离输入，并在超时后强杀。Agent Lab 会让可信协调器运行 `game-core` 和指标计算，再通过行协议与持久化 Bot 子进程通信，既避免每步启动 JVM，也不让候选代码接触隐藏种子或篡改评测结果。

Agent 层采用有界状态机，不让模型自由决定任意工具和流程。每个 Step 有幂等键，状态更新使用乐观锁。服务重启后根据最后成功 Step 恢复，重复调度不会重复评测。模型输出经过结构化校验，越权动作直接拒绝。

评测层采用固定地图、固定基准 Bot 和交换出生位置，公开集用于改进，隐藏集只在迭代结束后用于选择。最终版本必须合法移动率为 100%、延迟满足阈值，并通过隐藏集比较。这样即使模型声称「已优化」，系统也可能拒绝该版本。

## 高频追问

> 当前 Codex App 未暴露原生子 Agent 工具，因此本次按技能降级方案，分别使用基础追问、系统设计和怀疑论三种人格，只基于候选人 claim 包生成问题。问题与回答的分离度弱于独立子 Agent，后续可再进行一次真正的 source-blind 模拟面试。

### 问题 1：`game-core` 怎样保证在线对战和离线评测真的一致？ 〔基础追问〕

**推荐回答：**

不能只复制一份规则。我的方案是先用当前 `GameRules`、固定地图和录像建立回归测试，再把地图生成、回合推进、碰撞和结果判定抽到不依赖 Spring 的 `game-core`。在线 `Game` 和离线评测都调用同一接口。迁移时使用固定输入对比抽取前后的结果与录像，确保行为等价。

**面试官想考察：** 共享代码是否真正消除双实现，而不是换个目录复制。

**容易踩坑：** 只说「写单测」但讲不出测试输入、输出和等价性标准。

### 问题 2：为什么状态机用了数据库乐观锁，还需要幂等键？ 〔基础追问〕

**推荐回答：**

两者解决的问题不同。乐观锁保护状态转移，例如两个执行线程不能同时把 `EVALUATING` 推进到 `ANALYZING`。幂等键保护外部副作用，例如服务在评测完成但状态更新前崩溃，恢复后不能再次运行同一版本的 72 局比赛。状态条件和工具结果去重需要同时存在。

**面试官想考察：** 是否理解并发控制与副作用幂等的区别。

**容易踩坑：** 把乐观锁说成分布式锁，或者声称它能自动回滚外部进程。

### 问题 3：沙箱具体能防住什么，防不住什么？ 〔基础追问〕

**推荐回答：**

当前实现能通过独立 JVM 隔离主服务内存，限制网络、文件写入和外部命令，并通过超时强杀处理死循环。它不能像容器和 cgroup 一样可靠限制 CPU、内存、进程树和内核攻击面，而且 Java 8 `SecurityManager` 不是长期方案。因此我只把它定位为本地受控演示；生产化需要容器、资源配额和独立执行节点。

**面试官想考察：** 是否诚实理解安全边界。

**容易踩坑：** 说成「完全安全」或「和 Docker 一样」。

### 问题 4：任务量扩大 10 倍后，哪里最先成为瓶颈？ 〔系统设计〕

**推荐回答：**

最先受压的是批量评测的 CPU 和子进程数，其次是模型 API 并发与 MySQL Trace 写入。MVP 用全局有界线程池和单用户一个任务限制资源。规模增长后，我会把 Workflow 状态留在数据库，把评测拆到消息队列驱动的独立 Worker，并按 CPU 核数设置并发；Trace 可以批量落库或异步写入。

**面试官想考察：** 是否能从资源模型判断瓶颈，而不是直接堆中间件。

**容易踩坑：** 一上来回答 Kubernetes，却说不清当前机器的 CPU、进程和任务关系。

### 问题 5：固定评测集会不会让 Agent 过拟合？ 〔系统设计〕

**推荐回答：**

会，所以公开集和隐藏集必须分离。Agent 只能看到公开集聚合指标和少量失败轨迹，隐藏验证要等全部迭代结束后才运行，种子和运行结果都不进入 Agent 上下文。候选版本即使公开集提升，只要隐藏集表现不满足门禁就不能替换最佳版本。后续还可以轮换隐藏种子并增加多次重复实验。

**面试官想考察：** 是否理解模型评测的数据泄漏和泛化问题。

**容易踩坑：** 把「固定种子可复现」误说成「固定种子天然公平且不会过拟合」。

### 问题 6：为什么不让 LLM 每回合直接决定方向？ 〔系统设计〕

**推荐回答：**

实时对战每回合有严格延迟要求，而 LLM 调用存在网络延迟、成本和非确定性，会把模型可用性变成对战单点。我的分层是让 LLM 离线生成确定性 Java 策略，运行时只执行本地代码。这样既保留模型的创造能力，也让实时链路可测试、可复现和低成本。

**面试官想考察：** 是否能根据场景决定 LLM 应放在哪一层。

**容易踩坑：** 只回答「贵」，没有讲实时性和确定性。

### 问题 7：这和调用一次模型生成代码有什么本质区别？ 〔怀疑论〕

**推荐回答：**

一次生成只有输入和输出，无法验证代码是否可编译、是否安全、是否变强。Agent Lab 有持久化任务状态、受控工具、编译反馈、确定性评测、失败轨迹、版本选择和最多 3 轮改进。系统可以拒绝模型生成的候选版本，最终事实由评测器决定，不由模型自评。

**面试官想考察：** 项目是否真的具备 Agent 闭环，而不是 Prompt 包装。

**容易踩坑：** 用「用了 Tool Calling」代替说明工具、反馈和停止条件。

### 问题 8：你说胜率提升，怎么证明不是地图或随机性导致的？ 〔怀疑论〕

**推荐回答：**

我会固定地图种子、基准 Bot 版本和比赛配置，并交换双方出生位置。每个版本跑同一组公开与隐藏比赛，保留逐局结果和录像。简历只填写多次复现实验后稳定出现的指标，并同时报告样本数、基线和隐藏集结果。如果 V3 没有稳定优于 V1，就不写提升。

**面试官想考察：** 指标是否有基线、对照和复现条件。

**容易踩坑：** 只展示最好的一次运行，或者不说明比赛数量。

## 最容易答崩的题

1. **沙箱边界。**必须主动承认 Java 8 `SecurityManager` 与进程隔离的不足，并给出容器化升级方向。
2. **胜率提升的可信度。**必须说明固定种子、交换出生点、公开/隐藏划分、样本数和重复实验。
3. **在线与离线规则一致性。**必须能讲出从回归测试到共享 `game-core` 的迁移步骤。

## 可迁移的工程思想

### 1. 模型提出候选，系统验证事实

- **为什么好：** 避免让 LLM 负责数值计算、规则判断和质量自评。
- **如何迁移：** 在其他 Agent 项目中，把编译、测试、指标和权限检查放到确定性工具层。
- **最小产物：** 一个模型生成代码、测试工具验证、失败后修复的闭环。
- **面试表达：** 「我的 Agent 不直接宣称任务成功，完成条件由工程侧验证器决定。」

### 2. 有界 Tool Calling

- **为什么好：** 模型只能在当前状态允许的工具集合中行动，减少越权和无意义循环。
- **如何迁移：** 为每个状态定义允许动作、输入 Schema、最大调用次数和终止条件。
- **最小产物：** `AgentToolRouter` 与状态转移测试。
- **面试表达：** 「我把 Agent 自主性限制在明确的工具和状态边界内，而不是开放任意 Shell。」

### 3. 请求关联与过期响应隔离

- **为什么好：** 异步回调不能只依赖用户或会话身份。
- **如何迁移：** 为任务、迭代和工具调用增加层级标识，并在写状态前校验。
- **最小产物：** 迟到、重复和乱序回调测试。
- **面试表达：** 「我使用任务级和步骤级关联标识，旧响应即使到达也不能污染新状态。」

### 4. 公开反馈集与隐藏验证集

- **为什么好：** Agent 需要反馈才能改进，但不能无限看到最终验收数据。
- **如何迁移：** 将开发集反馈给模型，保留隐藏集做最终门禁。
- **最小产物：** 版本接受/拒绝规则和隐藏数据泄漏测试。
- **面试表达：** 「我把 Agent 改进和最终验收的数据隔离，避免针对评测集刷分。」

## 项目短板与风险

- Spring Boot 2.4.5、Java 8、Fastjson 和 Vue CLI 基线较旧，但首版不应混入大版本升级。
- `WebSocketServer` 仍承担较多职责并使用静态注入。
- 当前 `Game` 每局创建线程，`Consumer` 每次执行也创建线程，容量上限不清晰。
- Bot 沙箱缺少容器级 CPU、内存和进程数限制。
- 当前本地部署依赖 MySQL 和多进程，README 与脚本需要继续打磨。
- Agent Lab 尚未实现，因此未来 bullet 目前只能作为实施目标，不能提前写成已完成经历。
